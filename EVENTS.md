# Chessome: Event Catalog

## 1. Overview
Chessome utilizes an Event-Driven Architecture (EDA) to decouple services. When an action occurs in the API or Worker layers, a Domain Event is emitted to the Redis message broker. Other microservices (or the WebSocket Gateway) listen to these events to perform background actions.

## 2. Global Event Standard
All events adhere to the CloudEvents standard, ensuring standard metadata across all payloads.

```typescript
interface DomainEvent<T> {
  id: string; // UUID
  type: string; // Event Name
  source: string; // Microservice name (e.g., 'api', 'worker-3')
  timestamp: string; // ISO 8601
  data: T; // The payload
}
```

## 3. Event Catalog

### 3.1 Data Ingestion Events
- `GameImported`: Fired when a user manually uploads a PGN file. Triggers parsing.
- `GameSynced`: Fired when the OAuth background sync pulls a new game from Lichess/Chess.com. Triggers the Opening Classifier.
- `MasterGameIngested`: Fired when a high-level game is added to the public explorer database.

### 3.2 Analysis Pipeline Events
- `AnalysisQueued`: Fired when a FEN or Game is successfully written to the BullMQ queue. Frontend uses this to show a "Waiting in Queue" spinner.
- `AnalysisStarted`: Fired by a Worker Node when Stockfish is spawned. Frontend changes spinner to "Analyzing".
- `EvaluationProduced`: High-frequency event emitted by the Worker Node as the engine calculates (MultiPV lines). Pushed to Redis Pub/Sub for WebSockets.
- `EvaluationCompleted`: Fired when the engine reaches the target depth or time limit. Triggers database insertion.
- `GameReviewCompleted`: Fired when all 40 moves of a game have finished evaluation. Triggers the Accuracy/CAPS calculation worker.

### 3.3 System & Worker Events
- `WorkerOnline`: Fired when a Kubernetes worker pod boots and registers its capabilities.
- `WorkerOffline`: Fired gracefully during scaling down.
- `WorkerRecovered`: Fired when a stalled BullMQ job is successfully picked up by a new worker after a crash.
- `JobCancelled`: Fired if a user disconnects their WebSocket or manually halts the engine. The Scheduler kills the Stockfish process.

### 3.4 Caching Events
- `CacheHit`: Fired when the API instantly resolves a FEN evaluation from Redis without queueing. (Used for analytics/cache effectiveness monitoring).
- `CacheMiss`: Fired when an evaluation must be routed to the engine.

### 3.5 AI Integration Events
- `ExplanationRequested`: Fired when a user asks the AI a question.
- `ExplanationStreamChunk`: High-frequency event emitting SSE/WebSocket tokens from the LLM back to the user.
- `ExplanationCompleted`: Fired when the LLM finishes generating.

## 4. Architectural Use Case
**Decoupling the Frontend**: By standardizing `EvaluationProduced` and `EvaluationCompleted`, the React frontend (`useAnalysisStore`) simply listens to a WebSocket stream of these typed events. It does not know or care if the evaluation was generated by a Cloud Worker, a Local WASM thread, or pulled from the Redis Cache—the event shape is identical, making the UI highly resilient to backend architectural changes.
