# Chessome: Legal and Compliance Review

## 1. License Overview

Chessome employs a **Dual-License Strategy** to protect the core product from corporate exploitation while encouraging ecosystem growth.

### 1.1 The Core Platform (AGPLv3)
- **Applies to**: `apps/web`, `apps/api`, `apps/worker-analysis`.
- **Why**: The GNU Affero General Public License (AGPLv3) closes the "ASP loophole". If a commercial entity hosts a modified version of Chessome over a network (SaaS), they are legally required to open-source their modifications. This guarantees Chessome remains free.

### 1.2 The SDK & Plugins (MIT)
- **Applies to**: `packages/engine-adapters`, `packages/ui`, and future public NPM packages.
- **Why**: We want developers to use our highly optimized chess parsing and UI libraries in their own proprietary apps (e.g., a commercial iOS training app) without forcing them to open-source their entire codebase.

## 2. Third-Party Dependency Review

Every integrated library must be checked for copyleft infection risks.

### 2.1 Stockfish (GPLv3)
- **License**: GPLv3.
- **Compatibility**: AGPLv3 (Chessome) is compatible with GPLv3.
- **Implications**: Because we distribute Stockfish.wasm to the user's browser, our entire web client must remain open-source (which it is, under AGPL). The backend API merely communicates with Stockfish binaries via standard input/output (UCI over stdin/stdout), which under GPL is generally considered "mere aggregation" or communication at arm's length, but because our backend is AGPLv3 anyway, there are no licensing conflicts.

### 2.2 React, Next.js, NestJS, Prisma (MIT / Apache 2.0)
- **License**: MIT and Apache 2.0.
- **Compatibility**: Fully compatible with AGPLv3.
- **Implications**: We must retain their original copyright notices and license text in our repository (usually handled automatically by package managers or in a `CREDITS.txt` file).

## 3. Third-Party API Constraints

### 3.1 Lichess API
- **Terms of Service**: Requires a custom `User-Agent` string identifying the application. Strictly forbids abusing the API for heavy data scraping without permission.
- **Integration Limit**: Rate limits are strictly enforced (usually 1 request per second).
- **Compliance Strategy**: Our `LichessPlugin` implements a local queue/throttle. If a user requests a sync of 5,000 games, the plugin fetches them slowly via NDJSON streams, respecting the `429 Too Many Requests` headers.

### 3.2 Chess.com API
- **Terms of Service**: Published games (PGNs) are public domain, but player avatars and exact Elo ratings are subject to their specific API terms.
- **Integration Limit**: No official OAuth for third-party game syncing without partner status.
- **Compliance Strategy**: We rely strictly on their public `Published Data API`. We do not scrape HTML.

### 3.3 OpenAI / Anthropic API (BYOK)
- **Terms of Service**: Users supply their *own* API keys.
- **Compliance Strategy**: We act purely as a pass-through proxy. We do not store prompts or explanations for training purposes. We are not liable for the user exhausting their own API credits. We must ensure the UI clearly states that explanations are generated by a third-party AI and not Chessome itself.

## 4. Privacy & GDPR Compliance

- **Right to be Forgotten**: A dedicated API endpoint (`DELETE /v1/users/me`) immediately wipes PII (email, username, OAuth tokens) from the PostgreSQL database.
- **Data Anonymization**: Evaluated chess positions (FENs) and their Stockfish depths/scores are mathematically decoupled from the user. When a user deletes their account, the mathematical evaluations remain in the database (as they are universal truths of chess), but the link to the user's game is destroyed.
- **Cookies**: Strict separation of functional cookies (JWT Auth) vs analytics (none). No GDPR cookie consent banner is required because Chessome does not use third-party tracking cookies (Google Analytics, Facebook Pixel, etc.).
